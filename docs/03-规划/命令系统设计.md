# 命令系统设计

---

## 一、设计目标

本方案将命令系统收敛为一版**简洁、易懂、低 token 消耗**的设计，服务于以下目标：

1. AI 通过一个核心工具驱动前端表现
2. 支持一次输出多个命令
3. 不引入复杂 JSON 结构和过强的底层控制能力
4. 支持通过命令“说话”，从而在不中断 Agent 循环时驱动前端输出
5. 前端负责真正执行，AI 只负责表达控制意图

本命令系统的实施顺序与依赖关系，详见 [整体重构实施计划](./整体重构实施计划.md)。

---

## 二、最终原则

### 2.1 一个核心工具

AI 控制前端表现只通过一个核心工具：

- `perform_actions`

这个工具的职责不是“直接执行动作”，而是：

> 向前端发送一组按顺序执行的轻量命令。

### 2.2 一个专属外部控制工具

镜头等“外部控制”不与角色表现命令混在一起，单独使用专属工具：

- `control_camera`

这样可以让 AI 更清楚地区分：

- 我自己的表现
- 外部镜头/视角控制

### 2.3 AI 不直接控制状态机

以下状态由系统状态机自动维护，不交给 AI 直接控制：

- `idle`
- `listening`
- `thinking`
- `speaking`

AI 只负责：

- 说什么
- 用什么表情说
- 倾向使用什么动作
- 是否建议镜头切换

---

## 三、`perform_actions` 的最终输入格式

### 3.1 输入结构

`perform_actions` 的输入固定为：

```json
{
  "commands": [
    "emotion happy",
    "say happy | 你好呀",
    "wait 600",
    "motion wave"
  ]
}
```

### 3.2 为什么用列表而不是大字符串

- 一条命令一项，更稳定
- 更容易逐条校验和容错
- 更方便日志、调试和回放
- 更适合后续流式增量补充命令
- 相比复杂 JSON，仍然足够轻量

---

## 四、第一版允许的命令

第一版只开放 4 类命令给 AI：

### 4.1 `say`

格式：

```text
say <emotion> | <text>
```

示例：

```text
say happy | 你好呀
say sad | 唔……我有点担心你
say neutral | 那我们继续吧
```

语义：

- 显示文本
- 触发说话链路
- 将 `<emotion>` 作为该句的主表情
- TTS、字幕分段、音频播放由系统处理

约束：

- `say` 只保留两个参数：`emotion` 和 `text`
- 不允许在命令层直接指定音频 URL、动作 URL、复杂播放参数

### 4.2 `emotion`

格式：

```text
emotion <name>
```

示例：

```text
emotion happy
emotion neutral
emotion sad
```

语义：

- 立即切换当前表情
- 不自动说话
- 适合无台词反应

### 4.3 `motion`

格式：

```text
motion <name>
```

示例：

```text
motion wave
motion nod
motion think
```

语义：

- AI 表达“希望使用这个动作”
- 系统不一定立刻执行
- 系统会在合适的叙事时机播放

重点：

- `motion` 是动作意图，不是底层即时执行指令

### 4.4 `wait`

格式：

```text
wait <ms>
```

示例：

```text
wait 500
wait 1200
```

语义：

- 在叙事队列中插入停顿
- 用于控制表演节奏

---

## 五、明确不交给 AI 的东西

以下内容不开放给 AI：

- 状态机切换
- 实体增删
- 场景增删和场景编辑
- 角色加载
- 角色位置/缩放编辑
- 复杂镜头轨道
- 资源 URL 级控制

这些能力只允许：

- 系统内部控制
- UI 编辑器控制
- 专属管理工具控制

原因：

- 太偏编辑器能力
- 容易让 AI 过度控制前端
- 提高 prompt 与 tool schema 复杂度
- 不利于第一版稳定落地

## 六、执行语义

### 6.1 队列命令

以下命令进入叙事队列，按顺序执行：

- `say`
- `wait`

### 6.2 即时命令

以下命令立即生效：

- `emotion`

### 6.3 动作意图命令

以下命令不保证立刻执行，而是交给系统在合适时机播放：

- `motion`

推荐处理方式：

- `motion` 作为下一句 `say` 的动作意图
- 或作为当前叙事片段的动作候选

第一版建议采用更简单的规则：

> 最近一次 `motion` 命令，作为下一条 `say` 的优先动作。

示例：

```json
{
  "commands": [
    "motion wave",
    "say happy | 欢迎回来"
  ]
}
```

系统语义：

- 说“欢迎回来”时，优先配 `wave`

---

## 七、`control_camera` 的定位

`camera` 不再混在 `perform_actions` 里，而是独立工具：

- `control_camera`

第一版可以先保持简单结构化参数：

```json
{
  "action": "cut",
  "preset": "close"
}
```

或：

```json
{
  "action": "transition",
  "preset": "front",
  "durationMs": 500
}
```

这样能让 AI 理解：

- `perform_actions` 是角色表现
- `control_camera` 是镜头控制

## 八、前后端职责边界

### 8.1 AI 负责

- 选择说什么
- 选择说话时的表情
- 给出动作意图
- 给出停顿节奏
- 在需要时调用专属镜头工具

### 8.2 中间件 / 后处理负责

- 注入当前场景状态
- 把 `say` 的文本送入 TTS
- 生成语音片段与分段信息
- 发出 `custom events`
- 向前端传递额外运行状态

### 8.3 前端负责

- 消费 tool calls
- 消费 `custom events`
- 将命令转为真正的 VRM 控制
- 执行音频、字幕、表情、动作、镜头联动

---

## 九、前端执行链

```text
AI
 -> perform_actions(commands)
 -> Tool Call Event
 -> Frontend StreamAdapter
 -> Command Parser / Validator
 -> VRMDirector.dispatch()
 -> CommandExecutor
 -> Narrative / Emotion / Motion / Camera
 -> Store / R3F / Audio / Subtitle
```

对于 `say` 的语音链：

```text
AI 文本输出 / say 命令
 -> 中间件 / 后处理
 -> TTS
 -> custom event(vrm_segment)
 -> Frontend StreamAdapter
 -> NarrativeController
 -> 音频 / 字幕 / 表情 / 动作协同执行
```

## 十、解析与校验规则

### 10.1 后端 / 前端只接受有限语法

允许命令：

```text
emotion <name>
motion <name>
say <emotion> | <text>
wait <ms>
```

### 10.2 容错策略

- 空字符串跳过
- 非法命令跳过并记录日志
- 参数缺失跳过当前命令
- 不影响同一批次其他命令执行

### 10.3 归一化建议

建议解析后统一转换成内部对象：

```ts
type ParsedAction =
  | { type: 'emotion'; value: string }
  | { type: 'motion'; value: string }
  | { type: 'say'; emotion: string; text: string }
  | { type: 'wait'; ms: number };
```

这样 AI 输出保持轻量，执行层保持类型安全。

## 十一、推荐示例

### 11.1 纯说话

```json
{
  "commands": [
    "say happy | 你好呀"
  ]
}
```

### 11.2 带停顿

```json
{
  "commands": [

  ]
}
```

### 11.3 动作配合说话

```json
{
  "commands": [
    "motion wave",
    "say happy | 欢迎回来"
  ]
}
```

### 11.4 先切表情再说话

```json
{
  "commands": [
    "emotion sad",
    "say sad | 唔……这件事让我有点难过"
  ]
}
```

## 十二、最终结论

第一版命令系统正式收敛为：

- 一个角色表现工具：`perform_actions`
- 一个外部控制工具：`control_camera`
- 一个轻量输入结构：`commands: string[]`
- 四个 AI 可用命令：
  - `say`
  - `emotion`
  - `motion`
  - `wait`

这样可以保证：

- 简洁
- 易学
- token 成本低
- 可逐步扩展
- 不和状态机职责冲突
- 与前端 Director 执行架构兼容
