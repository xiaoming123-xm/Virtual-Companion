# 启动优化方案

> 本文档聚焦冷启动与首屏解阻塞，是 [系统性能与运行时升级方案](./系统性能与运行时升级方案.md) 的子集。

本文档用于约束 ATRI Chat 启动性能优化的目标架构、实施顺序与验收标准。目标不是让所有能力同时更快启动，而是让用户最先看到、最先使用到的部分尽快可用。

## 1. 问题定义

当前启动链路存在以下典型问题：

- 桌面壳在启动阶段同步等待后端健康检查完成，导致窗口可见时间与后端冷启动强耦合。
- FastAPI 在模块导入阶段一次性装配全部路由与依赖，ASR、Agent、LangChain 等重量级能力进入了冷启动关键路径。
- 前端在挂载 React 根节点前等待 API 地址初始化，进一步放大了白屏时间。
- AI 能力尚未区分“首屏必需能力”和“按需能力”，缺少统一的懒加载与后台预热策略。

## 2. 目标架构

整体改为“三阶段启动”：

1. Shell Ready
   Tauri 只负责窗口创建、目录准备、后端子进程拉起，不阻塞等待后端健康检查。

2. Control Plane Ready
   后端仅保证健康检查、配置解析、静态资源、轻量查询接口可用。

3. Capability Ready
   Agent、ASR、TTS 等重量级能力在首次使用时初始化，并允许在系统空闲时后台预热。

### 2.1 设计原则

- 首屏优先：冷启动路径只保留“窗口出现”和“基础 API 可用”所需能力。
- 能力分级：将系统能力拆分为控制面能力与执行面能力。
- 精确导入：避免通过聚合导出或类型导入触发整条重依赖链。
- 状态可观测：前端显式展示启动阶段，避免用户感知为卡死。

## 3. 分阶段实施

### 阶段 A：后端导入瘦身

目标：降低 `import main` 和 `/health` 可用前的冷启动成本。

实施点：

- 精简 `core/__init__.py`，仅保留轻量配置模型导出，不再聚合导出 `AgentCoordinator`、`ModelFactory`、Repository、Service。
- `core/dependencies.py` 改为函数内导入，避免模块加载阶段引入 LangGraph、Agent、Store。
- `core/models/factory.py` 将 LangChain/Qwen 相关导入下沉到真正创建模型时。
- `api/routes/messages.py`、`api/routes/providers.py`、`api/routes/asr.py` 去除通过类型标注触发的重导入。

### 阶段 B：能力懒加载

目标：让 ASR、Agent 等能力只在首次真正使用时付出初始化成本。

实施点：

- ASR 路由仅保留依赖接口，`sherpa_onnx` 在识别器初始化时再导入。
- 供应商模板列表等轻量后台操作直接依赖 `ModelFactory`，避免触发 `AgentCoordinator`。
- Agent 保持按需单例化，冷启动阶段不创建真实 Agent 实例。

### 阶段 C：首屏解阻塞

目标：让窗口尽快可见，并将“后端就绪”转换为前端可观察的启动状态。

实施点：

- Tauri 不再在 `setup()` 中等待后端健康检查。
- 前端根节点立即渲染启动页，异步完成 API 地址初始化与后端健康检查。
- 主应用只在 `API_CONFIG` 初始化完成且健康检查通过后挂载。

## 4. 验收指标

建议用以下指标作为阶段性验收：

- 窗口可见时间
- 前端首屏可交互时间
- 后端 `/api/v1/health` 首次返回 200 的时间
- 首次聊天请求延迟
- 首次 ASR 请求延迟

## 5. 本轮实施范围

本轮优化聚焦以下内容：

- 完成启动方案文档落库
- 落地后端导入瘦身
- 落地 ASR 与模型工厂的懒导入
- 将 Tauri 启动改为非阻塞模式
- 为前端增加启动页和健康检查门禁

本轮暂不处理：

- Checkpointer 的彻底延后初始化
- Agent 预热调度器
- 启动性能埋点上报系统

## 6. 后续演进建议

- 为启动阶段补充结构化耗时埋点，区分 Python 导入、数据库初始化、前端挂载、首个请求等耗时。
- 为 Agent、ASR、TTS 建立统一的能力注册表与状态机。
- 将控制面进程和执行面能力进一步拆分，为后续多模态与本地推理扩展做准备。
