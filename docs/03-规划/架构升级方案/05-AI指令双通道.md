# AI 指令双通道设计

---

## 一、核心洞察

AI 控制角色有两类根本不同的指令：

| | 表演指令（Performance）| 逻辑指令（Logic）|
|---|---|---|
| **例子** | 情感切换、手势动作、停顿 | 移动到某处、切镜头、操作物体 |
| **时序要求** | 必须与**语音对齐**（说到"你好"时挥手）| **与语音独立**（可以先移动再说话）|
| **格式偏好** | 内嵌文本标记（因为要对齐词语）| 结构化工具调用（需要参数验证和返回值）|
| **可否中途执行** | 不可，必须等到 TTS 流到此处 | 可以，立即执行 |

**结论：** 必须双通道，不能只用一种。

---

## 二、双通道架构

```
┌──────────────────────────────────────────────────────────────┐
│                       AI Agent (后端)                         │
│                                                              │
│  输出两类数据：                                               │
│                                                              │
│  1️⃣ 工具调用通道（Tool Channel）                              │
│     perform_actions({                                        │
│       actions: [                                             │
│         { type: "camera", mode: "front", dist: "close" },   │
│         { type: "gaze", target: "cup_blue" },               │
│       ]                                                      │
│     })                                                       │
│     ↓  通过 useStream.toolCalls 到达前端                     │
│     ↓  立即执行，不排队                                      │
│                                                              │
│  2️⃣ 文本流通道（Text Stream Channel）                         │
│     流式输出带内嵌标记的文本：                                │
│     "<emo v='happy'/>你好！<act v='wave'/>很高兴见到你。"    │
│     ↓  后端 TTS 分段 → onCustomEvent 推送                   │
│     ↓  进入叙事队列，按顺序播放                              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
                    │                        │
                    ↓                        ↓
         ┌─────────────────┐     ┌──────────────────────┐
         │  CommandAdapter  │     │   CommandAdapter      │
         │  (Tool→Command)  │     │   (Markup→Command)    │
         └────────┬────────┘     └──────────┬───────────┘
                  │                          │
                  ↓                          ↓
         ┌─────────────────────────────────────────────┐
         │          VRMDirector.dispatch()              │
         │             统一命令类型                      │
         └─────────────────────────────────────────────┘
```

## 三、通道1：工具调用 — `perform_actions`

### 3.1 后端工具定义

```python
class PerformActions(BaseModel):
    """控制虚拟角色的身体动作、镜头和场景。
    当你需要在说话之前或独立于说话时执行物理动作时使用。"""

    actions: list[ActionItem] = Field(
        description="要执行的动作列表，按顺序执行"
    )

class ActionItem(BaseModel):
    type: Literal[
        "camera",     # 切镜头
        "move",       # 移动到目标
        "gaze",       # 注视目标
        "motion",     # 播放骨骼动作
        "emotion",    # 设置情感
        "sfx",        # 播放音效
        "wait",       # 等待N秒
        "scene",      # 场景操作
    ]
    # 通用参数（各类型使用不同子集）
    target: str | None = None       # 目标ID（move/gaze）
    mode: str | None = None         # 模式（camera: front/ots/fpv）
    dist: str | None = None         # 距离（camera: close/mid/far）
    gait: str | None = None         # 步态（move: walk/run）
    name: str | None = None         # 名称（motion/emotion）
    intensity: float = 1.0          # 强度（emotion: 0~1）
    duration_ms: int | None = None  # 持续时间（wait）
    loop: bool = False              # 是否循环（motion）
    url: str | None = None          # URL（sfx: 音效URL）
```

### 3.2 前端适配器

```typescript
// CommandAdapter: Tool → VRMCommand
function adaptToolActions(actions: ActionItem[]): VRMCommand[] {
  return actions.map(action => {
    switch (action.type) {
      case 'camera':
        return { type: 'CAMERA_CUT', mode: action.mode, dist: action.dist };
      case 'emotion':
        return { type: 'SET_EMOTION', emotion: action.name, intensity: action.intensity };
      case 'motion':
        return { type: 'PLAY_MOTION', motion: action.name, loop: action.loop };
      case 'gaze':
        return { type: 'SET_GAZE', target: action.target };
      case 'sfx':
        return { type: 'PLAY_SFX', url: action.url };
      case 'wait':
        return { type: 'WAIT', ms: action.duration_ms };
      // ... 更多类型
    }
  });
}
```

## 四、通道2：文本内嵌标记

### 4.1 标记语法

用于**必须与语音同步**的指令：

```
旧语法：[State:happy][Action:wave]你好！                    ← 兼容
新语法：<emo v="happy"/>你好！<act v="wave"/>很高兴见到你。  ← 推荐

完整标签集：
<emo v="happy" i="0.8"/>     情感（v=名称, i=强度，可选）
<act v="wave"/>               动作（v=名称或分类）
<pau ms="1500"/>              停顿（ms=毫秒）
```

### 4.2 为什么用 XML 而不是方括号

1. XML 是自封闭的（`/>`），解析更可靠
2. 支持多属性（`v="happy" i="0.8"`）
3. LLM 对 XML 标记的遵从度比自定义格式更高
4. 未来可扩展更多属性

### 4.3 前端标记解析器

```typescript
// CommandAdapter: Markup → VRMCommand

function parsePerformanceTags(markedText: string): {
  text: string;
  commands: VRMCommand[];
} {
  const commands: VRMCommand[] = [];

  // 正则匹配标签
  const tagRegex = /<(emo|act|pau)\s+([^/]*)\s*\/>/gi;
  let match;

  while ((match = tagRegex.exec(markedText)) !== null) {
    const [, tag, attrs] = match;
    const parsed = parseAttributes(attrs);

    switch (tag) {
      case 'emo':
        commands.push({
          type: 'SET_EMOTION',
          emotion: parsed.v,
          intensity: parseFloat(parsed.i || '1.0'),
        });
        break;
      case 'act':
        commands.push({
          type: 'PLAY_MOTION',
          motion: parsed.v,
        });
        break;
      case 'pau':
        commands.push({
          type: 'WAIT',
          ms: parseInt(parsed.ms || '1000'),
        });
        break;
    }
  }

  // 移除标签得到纯文本
  const text = markedText.replace(/<[^>]+\/>/g, '').trim();

  return { text, commands };
}

function parseAttributes(attrStr: string): Record<string, string> {
  const result: Record<string, string> = {};
  const regex = /(\w+)=["']([^"']*)["']/g;
  let match;
  while ((match = regex.exec(attrStr)) !== null) {
    result[match[1]] = match[2];
  }
  return result;
}
```

### 4.4 后端处理流程

1. LLM 输出带标记的文本
2. 后端按句子分段
3. 每段发给 TTS（VRMService 去除标记后合成）
4. 通过 `onCustomEvent` 推送 `vrm_segment`，包含 `marked_text` + `audio_url`
5. 前端 StreamAdapter 收到后解析标记，dispatch 给 Director

## 五、两个通道如何协作

一次完整的 AI 回复示例：

```
Step 1: AI 调用 perform_actions (Tool Channel)
        → actions: [{ type: "camera", mode: "front", dist: "close" }]
        → 前端立即切近景

Step 2: AI 输出文本流 (Text Stream Channel)
        → "<emo v='happy'/>你好！<act v='wave'/>很高兴见到你。"
        → 后端 TTS 分段
        → onCustomEvent vrm_segment
        → StreamAdapter → Director → 叙事队列 → 有序播放

Step 3: AI 再次调用 perform_actions (Tool Channel)
        → actions: [{ type: "camera", mode: "ots", dist: "mid" }]
        → 切回远景
```

**关键：Tool 通道立即执行，Text 通道有序排队，两者互不干扰。**

## 六、兼容性策略

### 阶段1（前端先行）
- CommandAdapter 同时支持旧 `[State:xxx]` 和新 `<emo/>` 两种格式
- 新架构通过 adapter 层消化旧格式

### 阶段2（后端配合）
- 后端注册 `perform_actions` 工具
- 系统提示词更新，教 AI 使用新标签语法
- AI 开始输出 `<emo/>` `<act/>` `<pau/>` 标签

### 阶段3（全面 Agent 化）
- 后端每轮注入场景上下文
- AI 可以调用 `perform_actions` 在说话前后编排表演
