# AI 上下文注入机制

> AI 需要知道"我有什么能力、我在哪、周围有什么"才能做出合理的决策。
> 本文档描述了 AI 获取上下文的完整机制。

---

## 一、上下文分类

AI 的上下文分为两大类：

| 类别 | 内容 | 变化频率 | 注入方式 |
|---|---|---|---|
| **静态/半静态** | 工具列表、可用表情、可用动作、指令 Prompt | 角色/模式切换时变化 | Agent 初始化绑定 + System Prompt |
| **动态** | 实时世界状态（位置、情感、注视、附近物体） | 每轮对话变化 | 前端全量快照 + 后端中间件注入 |

---

## 二、静态上下文：Agent 初始化时绑定

### 2.1 工具列表 (Tools)

- **实现方式**：LangChain `bind_tools(tools)` 绑定到 Agent
- **注入时机**：Agent 创建时，或角色/模式切换时重新绑定
- **更新机制**：**全量更新**。`filter_tools_by_mode` 中间件根据当前模式动态过滤

```python
# core/middleware/dynamic_tools.py

# VRM 模式下的工具集
vrm_tools = [perform_actions, memory_save, memory_search]

# Text 模式下的工具集
text_tools = [memory_save, memory_search, web_search]

# 动态绑定
model_with_tools = model.bind_tools(
    vrm_tools if mode == 'vrm' else text_tools
)
```

### 2.2 指令 Prompt（Command Prompt）

通过 `CharacterPromptMiddleware` 注入到 System Message 中，教 AI 如何使用标签和工具。

```markdown
## 角色指令

你是 ATRI，一个活泼可爱的AI助手。

### 表演指令
在回复中使用以下标签控制你的表现：
- `<emo v="happy"/>` — 切换表情（在说出对应内容之前插入）
- `<act v="wave"/>` — 播放动作
- `<pau ms="1000"/>` — 停顿指定毫秒

### 可用表情列表
neutral, happy, sad, angry, surprised, excited, relaxed, confused, embarrassed

### 可用动作列表
wave, bow, nod, shake_head, think, point, clap, sit_down, stand_up, walk

### 工具使用
当需要在说话之前或独立于说话时执行动作，使用 `perform_actions` 工具：
- 切镜头：`{ "type": "camera", "mode": "front", "dist": "close" }`
- 注视目标：`{ "type": "gaze", "target": "cup_blue" }`
- 移动到目标：`{ "type": "move", "target": "desk_01" }`
- 操作场景物体：`{ "type": "attach", "objectId": "cup_01", "bone": "RightHand" }`
```

### 2.3 表情列表和动作列表的来源

| 数据 | 来源 | 注入方式 |
|---|---|---|
| **可用表情** | VRM 模型的 BlendShape 列表（后端 `VRMService` 解析） | 写入角色的 System Prompt |
| **可用动作** | 后端 `GET /api/v1/motions?character_id=xxx` 返回 | 写入角色的 System Prompt |
| **背景列表** | 后端 `GET /api/v1/assets/backgrounds` 返回 | 写入角色的 System Prompt（如果 AI 需要切背景） |

**注入时机**：角色加载时，后端中间件从数据库拉取这些列表并拼入 System Prompt。
**更新机制**：**全量注入**（每轮对话都带完整列表，保证 AI 不会"忘记"能力）。

## 三、动态上下文：每轮对话的世界快照

### 3.1 流程总览

```
用户点击发送
    │
    ├─→ useAgentStream.submitWithContext(content)
    │     │
    │     ├─→ 从 VRMWorldStore 获取全量快照
    │     │     entities, engagement, emotion, gaze, posture,
    │     │     camera, sceneConfig, characterConfig
    │     │
    │     └─→ stream.submit({
    │           messages: [{ type: "human", content }],
    │           vrm_context: { ...全量快照... }
    │         })
    │
    │                    ┌─── 后端 ───┐
    │                    │             │
    │                    ↓             │
    │            vrm_context.py 中间件拦截
    │            ├─→ 从 request 中提取 vrm_context
    │            ├─→ 格式化为 System Message
    │            └─→ 插入到 messages[-1] (HumanMessage) 之前
    │                    │             │
    │                    ↓             │
    │            AI 看到完整的上下文    │
    │            做出决策并回复         │
    │                    └─────────────┘
    │
    └─→ AI 回复改变了前端状态（情感/动作/注视等）
        ↓ 下一轮对话时，新状态会再次作为快照注入
```

### 3.2 前端快照构建

```typescript
// hooks/useAgentStream.ts

function buildVRMContext(): VRMContext {
  const state = useVRMWorldStore.getState();

  // 从 entities 中提取可见实体列表（供 AI 感知"附近有什么"）
  const visibleEntities = Object.values(state.entities)
    .filter(e => e.visible && !e.isMain)
    .map(e => ({
      id: e.id,
      name: e.id,        // TODO: 从资产库获取人类可读名称
      type: e.type,
      position: e.position,
      parentId: e.parentId,
    }));

  return {
    character: {
      engagement: state.engagement,
      emotion: state.emotion.current,
      posture: state.posture,
      gaze: state.gaze.target,
    },
    scene: {
      nearby_objects: visibleEntities,
      background: state.sceneConfig.background.type,
    },
    camera: state.camera,
  };
}
```

### 3.3 后端注入

```python
# core/middleware/vrm_context.py

@wrap_model_call
async def inject_vrm_context(request, handler):
    vrm_state = request.runtime.context.get('vrm_context')

    if vrm_state:
        # 构建场景描述文本
        scene_lines = [
            "[当前 VRM 场景状态]",
            f"- 交互状态: {vrm_state.get('character', {}).get('engagement', 'unknown')}",
            f"- 当前表情: {vrm_state.get('character', {}).get('emotion', 'neutral')}",
            f"- 角色姿态: {vrm_state.get('character', {}).get('posture', 'standing')}",
            f"- 注视目标: {vrm_state.get('character', {}).get('gaze', 'user')}",
        ]

        # 附近物体
        objects = vrm_state.get('scene', {}).get('nearby_objects', [])
        if objects:
            scene_lines.append("- 附近物体:")
            for obj in objects:
                parent_info = f" (手持)" if obj.get('parentId') else ""
                scene_lines.append(
                    f"  - {obj['id']}: {obj.get('name', obj['id'])} "
                    f"位置={obj.get('position', 'unknown')}{parent_info}"
                )
        else:
            scene_lines.append("- 附近物体: 无")

        # 作为 system message 注入
        scene_msg = "\n".join(scene_lines)
        msgs = list(request.messages)
        msgs.insert(-1, {"role": "system", "content": scene_msg})
        return await handler(request.override(messages=msgs))

    return await handler(request)
```

## 四、为什么是"全量快照"而不是"增量更新"

| 方案 | 优点 | 缺点 |
|---|---|---|
| **全量快照（采用）** | 容错性强（某轮失败不影响下轮）；AI 无需记忆；实现简单 | 每轮多几百 Token（约 200~500） |
| 增量更新 | Token 数更少 | AI 需自行累加状态，容易幻觉；一轮丢失则后续全错 |

**结论**：Token 成本可忽略（200 Token 约 $0.0001），但容错性和可靠性的收益巨大。

## 五、闭环：AI 的决策如何反馈到下一轮上下文

```
轮次 N:
  AI 看到：emotion=neutral, gaze=user
  AI 决定：<emo v='happy'/> + perform_actions([gaze cup_blue])
  前端执行：emotion→happy, gaze→cup_blue

轮次 N+1:
  前端发送快照：emotion=happy, gaze=cup_blue
  AI 看到：emotion=happy, gaze=cup_blue
  AI 知道"我上一轮变开心了，正在看杯子"
  → 做出连贯的后续决策
```

这形成了一个**感知-决策-执行-反馈**的闭环，使 AI 的行为具有连续性和一致性。

## 六、性能与安全考量

### 6.1 Token 开销评估

| 注入内容 | 估计 Token 数 | 频率 |
|---|---|---|
| System Prompt（静态） | 300~800 | 每轮（LLM 需要） |
| 表情/动作列表 | 100~200 | 每轮（含在 System Prompt 中） |
| 世界状态快照 | 200~500 | 每轮 |
| **总额外开销** | **~500** | 相比省去 1 次工具调用（节约 1 次 LLM 推理），净收益为正 |

### 6.2 安全过滤

- 前端构建快照时**不发送敏感信息**（如 API Key、内部 URL）
- 后端中间件对 `vrm_context` 做白名单校验，只接受已知字段
- 实体列表只发送 ID、名称和位置，不发送模型 URL
