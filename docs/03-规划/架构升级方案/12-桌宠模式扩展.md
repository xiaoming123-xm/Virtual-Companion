# 桌宠模式扩展方案 (Desktop Pet Mode)

> 基于 Tauri 多窗口架构与轻量独立渲染的扩展方案。

---

## 一、方案概述

得益于本作的**三层分离架构**，我们可以将渲染层（R3F）完全从主界面的复杂逻辑（消息流、对话、工具卡片）中剥离出来。
利用 Tauri 原生的多窗口支持，我们可以创建一个完美的**桌面宠物 (Desktop Pet)**。

### 桌宠模式的核心需求：
1. **窗口全透明**（无背景，仅渲染角色和道具）。
2. **无边框、可置顶**。
3. **与主窗口共享状态**（动作、表情、语音）。
4. **低性能消耗**（背景闲置时 GPU 降频）。

---

## 二、架构设计：双窗口分离

我们采用**控制端（主窗口）+ 表现端（桌宠窗口）**的架构。

```
┌─────────────── Tauri 宿主进程 (Rust) ──────────────┐
│                                                  │
│  ┌─── 主窗口 (main.html) ─────────────┐          │
│  │                                    │          │
│  │  useAgentStream                    │          │
│  │  VRMDirector (大脑)                 │          │
│  │  ChatView (聊天 UI)                │          │
│  │                                    │          │
│  └───────┬────────────────────────────┘          │
│          │                                       │
│          │ Tauri IPC 事件 (emit / listen)        │
│          │ 或局域网 WebServer 状态同步           │
│          ↓                                       │
│  ┌─── 桌宠窗口 (pet.html) ────────────┐          │
│  │  [透明, 无边框, 置顶]                │          │
│  │                                    │          │
│  │  VRMPreviewer (四肢)                │          │
│  │  + 独立的 Zustand Store             │          │
│  │    (仅接收渲染快照)                  │          │
│  └────────────────────────────────────┘          │
└──────────────────────────────────────────────────┘
```

> **为何不使用唯一的 Store？**
> 因为 Tauri 每个窗口是一个独立的 WebView 进程，它们之间的 JavaScript 内存是不共享的。

---

## 三、Tauri 配置与原生支持

在 `tauri.conf.json` 中配置基础系统窗口：

```json
{
  "tauri": {
    "windows": [
      {
        "label": "main",
        "title": "ATRI Chat",
        "fullscreen": false,
        "width": 1200,
        "height": 800
      },
      {
        "label": "pet",
        "title": "ATRI Pet",
        "transparent": true,
        "decorations": false,
        "alwaysOnTop": true,
        "visible": false,
        "width": 300,
        "height": 450
      }
    ]
  }
}
```

**关键特性支持：**
- **背景透明**：HTML `<body>` 和 React-Three-Fiber `<Canvas alpha={true}>` 设置透底，结合 Tauri `transparent: true`，角色将悬浮于 Windows/macOS 桌面之上。
- **防止干扰**：通过 Tauri API 可启用 `setIgnoreCursorEvents(true)`（可选），实现鼠标穿透操作（点击角色会穿透到后面的桌面图标）。

## 四、状态同步机制：事件桥接 (Event Bridge)

因为主窗口（Director 所在）和桌宠窗口（纯渲染）是两个独立的 WebView 进程，我们需要通过 Tauri 的 IPC 事件总线进行**单向状态同步**。

### 1. 主窗口发送同步快照

在 `VRMDirector` 的末端，添加一个监听器，将关键渲染状态广播出去：

```typescript
// main-window: 同步服务
import { emit } from '@tauri-apps/api/event';

useVRMWorldStore.subscribe(
  // 提取需要同步的精简状态
  (state) => ({
    emotion: state.emotion.current,
    motionUrl: state.motion.url,
    gazeTarget: state.gaze.target
  }),
  (snapshot) => {
    // 通过 Tauri IPC 广播给 pet 窗口
    emit('sync_to_pet', snapshot);
  }
);
```

### 2. 桌宠窗口接收并驱动渲染

桌宠窗口极度轻量，没有 `useStream`，没有 Director，只有一个简易 Store 接收数据并喂给 `VRMPreviewer`。

```typescript
// pet-window: 接收与渲染
import { listen } from '@tauri-apps/api/event';

function PetApp() {
  const [petState, setPetState] = useState({ emotion: 'neutral', motionUrl: null });

  useEffect(() => {
    const unlisten = listen('sync_to_pet', (event) => {
      setPetState(event.payload);
    });
    return () => unlisten.then(f => f());
  }, []);

  return (
    // 使用第七章设计的轻量预览组件
    <VRMPreviewer 
      modelUrl="/models/atri.vrm" 
      motionUrl={petState.motionUrl}
      // 此处可添加透传表情的 prop
    />
  );
}
```

## 五、音频与同步降级方案

**桌宠音频播放问题：**
- 如果用户开着主窗口和桌宠窗口，两边同时播放音频会产生回音。
- **策略**：桌宠窗口**默认静音** (Mute AudioNode)。桌宠只需要读取音频分频器（AnalyserNode）的数据来驱动**口型 (LipSync)**即可，发声由主窗口负责。

**性能降级 (Throttle)：**
- 桌宠长期挂机时，不能让 GPU 持续拉满。
- **策略**：监听系统空闲状态。当 `engagement === 'idle_bored'` 且持续时间超过 5 分钟，切换 R3F `frameloop="demand"`，仅在特定动作（如眨眼、呼吸循环点）手动触发 `invalidate()` 重新渲染一帧，实现"睡眠"态的极低功耗。
