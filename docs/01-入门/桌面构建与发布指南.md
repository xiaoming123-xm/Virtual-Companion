# 桌面构建与发布指南

本文档是 ATRI Chat 桌面端构建、打包与发布的统一入口，同时约定发布产物、运行时目录、命名规范与版本流程。

适用范围：

- Windows 桌面端正式发布
- Tauri 壳层 + Python Sidecar 后端
- `Setup.exe` 与便携版双产物策略

若当前代码实现与本文不一致，以本文作为后续收敛方向。

## 1. 发布产物

ATRI Chat 对外正式发布两种产物：

| 产物类型 | 面向对象 | 说明 |
| :--- | :--- | :--- |
| `Setup.exe` | 普通用户 | 标准安装包，适合长期使用与常规升级 |
| `Portable.zip` | 高级用户 / 迁移场景 | 绿色免安装版，所有数据与程序同目录存放 |

补充约束：

- 后端 Sidecar 单独构建产物仅作为内部开发产物保留，不对外发布。
- 安装版与便携版必须共享同一套核心业务代码与配置协议，不允许出现功能分叉。

---

## 2. 运行模式

桌面端运行模式统一分为三类：

| 模式 | 说明 |
| :--- | :--- |
| `development` | 本地源码开发模式 |
| `installed` | 安装版运行模式 |
| `portable` | 便携版运行模式 |

运行模式由程序根据实际运行情况自动判定，而不是通过环境变量注入。

明确判定规则：

- 源码直接运行 Python 或 Tauri 调试构建时：视为 `development`
- 打包产物运行时，如果应用根目录下存在 `portable.mode`：视为 `portable`
- 打包产物运行时，若不存在 `portable.mode`：视为 `installed`

开发模式沿用便携版目录结构。

## 3. 目录策略

### 3.1 开发环境

| 项目 | 路径 |
| :--- | :--- |
| `app_root` | 仓库根目录 |
| `data_root` | `<repo>/data` |
| `log_root` | `<repo>/data/logs` |

### 3.2 安装版

安装版采用“程序目录与数据目录分离”的常规桌面应用策略。

| 项目 | 路径 |
| :--- | :--- |
| `app_root` | 安装目录 |
| `data_root` | `%LOCALAPPDATA%/ATRI-Chat/data` |
| `log_root` | `%LOCALAPPDATA%/ATRI-Chat/logs` |

设计原则：

- 安装目录只存放程序文件，不写运行时数据。
- 用户数据、数据库、模型、缓存、日志均写入 `LOCALAPPDATA`。
- 不使用 `%APPDATA%` 作为主要数据目录，因为本项目存在较大体积的数据与模型，不适合漫游配置目录。

### 3.3 便携版

便携版采用“程序与数据同目录”策略，但运行时数据统一归档到 `data/` 下。

| 项目 | 路径 |
| :--- | :--- |
| `app_root` | 应用目录 |
| `data_root` | `<app>/data` |
| `log_root` | `<app>/data/logs` |

推荐结构：

```text
ATRI Chat.exe
portable.mode
README.txt
binaries/
data/
  sqlite/
  models/
  assets/
  memory/
  logs/
config/
```

设计原则：

- 解压即可运行，无需安装。
- 所有用户数据均随应用目录迁移。
- 便携版不依赖外部注册表状态、不依赖安装器卸载信息。

## 4. 运行时契约

跨进程只保留一个必要环境变量：

| 环境变量 | 作用 |
| :--- | :--- |
| `ATRI_BACKEND_PORT` | 后端监听端口 |

实现要求：

- 运行模式与目录策略由程序自身判定，不再由环境变量控制
- Rust 与 Python 使用一致的判定规则
- Python 后端根据实际运行位置自行推导 `app_root`、`data_root`、`logs_root`

## 5. 组件职责

### 5.1 Tauri 壳层

负责：

- 判定运行模式
- 计算根目录、数据目录、日志目录
- 选择并分配后端端口
- 启动 Python Sidecar
- 传递 `ATRI_BACKEND_PORT`
- 执行健康检查
- 在窗口关闭或应用退出时回收后端进程

不负责：

- 业务数据路径二次推导
- 数据库结构判断
- 应用业务配置解释

### 5.2 Python 后端

负责：

- 根据实际运行状态推导运行模式和目录
- 读取 `ATRI_BACKEND_PORT`
- 初始化目录、日志、数据库、模型路径
- 暴露 API 与业务能力

不负责：

- 猜测是否为安装版或便携版
- 依据 `Program Files` 或 `AppData` 位置反推运行模式

### 5.3 构建脚本

负责：

- 以 PyInstaller `onedir` 形式打包 Python 后端 Sidecar
- 构建前端与 Tauri 应用
- 整理安装版与便携版产物
- 生成发布文件名与校验文件

不负责：

- 运行时环境推断
- 应用版本的手工多点维护

## 6. 发布命名规范

展示名与分发名分离：

- 展示名：`ATRI Chat`
- 分发名：`atri-chat`

文件命名原则：

- 全小写
- 使用 ASCII 字符
- 使用 `-` 分隔
- 包含产品名、版本号、平台、架构、产物类型

正式发布资产命名：

- `atri-chat-1.0.1-windows-x64-setup.exe`
- `atri-chat-1.0.1-windows-x64-portable.zip`
- `atri-chat-1.0.1-sha256sums.txt`

其他约定：

- Git Tag 使用 `v1.0.1`
- Release 标题使用 `ATRI Chat v1.0.1`
- 文件名中版本号不带 `v` 前缀

## 7. 版本迭代规范

### 7.1 单一版本源

项目版本单一事实源定义为根目录的 `pyproject.toml`。

### 7.2 版本同步目标文件

发布前必须同步以下文件：

- `pyproject.toml`
- `frontend/package.json`
- `frontend/src-tauri/tauri.conf.json`
- `frontend/src-tauri/Cargo.toml`

建议增加版本同步脚本，例如：

- `scripts/bump_version.py`

### 7.3 构建前校验

构建正式发布产物前，必须校验：

- 上述版本文件完全一致
- 锁文件与 `pyproject.toml` 一致
- 产物命名符合本文约定

### 7.4 语义化版本建议

- `patch`：Bug 修复、打包流程修复、性能优化、小型重构
- `minor`：兼容性新增功能
- `major`：不兼容变更、数据迁移、目录策略调整

### 7.5 数据版本与应用版本分离

数据库结构、配置迁移、缓存格式不直接绑定应用版本号，应维护独立的 schema/version 机制。

## 8.1 安装器与卸载器约束

安装包遵循“标准安装流程 + 轻量卸载扩展”的原则。

安装器当前约束：

- 仅发布 `NSIS` 安装包
- 使用 `currentUser` 安装模式
- 默认跟随系统语言，不额外弹出语言选择器
- 安装器仅负责安装程序文件，不处理用户数据目录

卸载器当前约束：

- 卸载前尝试结束 `ATRI Chat.exe` 与后端 sidecar 进程
- 卸载前询问是否删除 `%LOCALAPPDATA%/ATRI-Chat`
- 默认选项为“保留用户数据”
- 仅在用户明确确认后删除本地数据目录

实现入口：

- Tauri 安装器配置位于 [frontend/src-tauri/tauri.conf.json](/C:/Users/24352/Desktop/src/atri-chat/frontend/src-tauri/tauri.conf.json)
- NSIS 轻量扩展位于 [frontend/src-tauri/nsis/custom_settings.nsh](/C:/Users/24352/Desktop/src/atri-chat/frontend/src-tauri/nsis/custom_settings.nsh)

## 9. 启动流程规范

Tauri 启动时执行顺序：

1. 判定运行模式
2. 计算目录并创建目录
3. 分配后端端口
4. 启动后端进程
5. 轮询健康检查接口
6. 健康检查成功后进入正常前端工作态
7. 退出时回收后端进程

后端启动时执行顺序：

1. 根据实际运行位置判定模式与目录
2. 读取 `ATRI_BACKEND_PORT`
3. 初始化日志与目录
4. 初始化数据库与其他核心组件
5. 对外提供 API
